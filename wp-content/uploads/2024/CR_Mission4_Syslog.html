Contexte et objectif de la mission
Pourquoi centraliser les journaux Syslog ?

Sans centralisation, les journaux sont stockés localement sur chaque équipement — mémoire limitée, perte en cas de redémarrage, consultation équipement par équipement.

Avec un serveur Syslog central, tous les événements réseau (erreurs, connexions, changements d'état) sont regroupés en un seul endroit, visibles en temps réel.

Objectifs : faciliter le diagnostic des incidents, assurer la traçabilité et permettre à tout administrateur de reprendre l'exploitation en cas d'absence.

Principe et niveaux de sévérité Syslog
Comment fonctionne Syslog ?

Syslog est un protocole standard (UDP port 514) qui permet aux équipements d'envoyer leurs messages de journalisation vers un serveur central.

Chaque message a un niveau de sévérité de 0 (urgence maximale) à 7 (débogage). La commande logging trap définit le niveau minimum à envoyer.

NiveauNomDescriptionUtilisé ici
0EmergenciesSystème inutilisable—
1AlertsAction immédiate requise—
2CriticalConditions critiques—
3ErrorsErreurs (lien down, etc.)—
4WarningsAvertissements—
5InformationalMessages d'information — niveau choisi dans ce TP✅ logging trap informational
6NotificationsConditions normales mais significatives—
7DebuggingMessages de débogage (très verbeux)—

Le niveau informational (5) capture les niveaux 0 à 5 — suffisant pour surveiller les événements importants (changements d'état, redémarrages, connexions) sans être trop verbeux comme le niveau debug.

Configuration Syslog sur les équipements Cisco
Même configuration sur les 4 équipements (avec une différence pour l'interface source)

Toutes les commandes sont identiques sur le routeur et les 3 switches — seule la commande logging source-interface diffère :

Routeur : g0/0 (interface physique LAN) | Switches : vlan 1 (interface de management).

Capture 1 — Configuration Syslog complète sur chaque équipement Cisco
Équipements Cisco — Syslog
! Accès via PuTTY Serial COM1 9600 baud
Switch> enable
Switch# configure terminal

! 1. Horodatage précis des logs (avec millisecondes)
Switch(config)# service timestamps log datetime msec

! 2. Déclarer l'adresse IP du serveur Syslog
Switch(config)# logging host 172.17.10.20

! 3. Niveau de journalisation : informational (niveaux 0 à 5)
Switch(config)# logging trap informational

! 4. Activer Syslog
Switch(config)# logging on

! 5. Buffer local de 64 Ko pour stocker les logs en mémoire
Switch(config)# logging buffered 64000

! 6a. Interface source — SUR LE ROUTEUR uniquement :
Router(config)# logging source-interface g0/0

! 6b. Interface source — SUR LES SWITCHES uniquement :
Switch(config)# logging source-interface vlan 1

! 7. Désactiver les logs sur la console (évite les interruptions pendant les configs)
Switch(config)# no logging console

Switch(config)# exit
Switch# write memory     ! Sauvegarder — CRUCIAL sur matériel réel
service timestamps log datetime msec
Pourquoi ? Ajoute la date et l'heure précise (avec ms) à chaque message de log. Sans ça, les logs arrivent sans timestamp → impossible de corréler les événements entre équipements.
logging host 172.17.10.20
Pourquoi ? Désigne l'adresse IP du serveur Syslog vers lequel envoyer les logs. Les messages sont transmis en UDP sur le port 514 — le serveur rsyslog écoute sur ce port.
logging trap informational
Pourquoi ? Définit le seuil de sévérité des messages envoyés au serveur distant. Le niveau 5 (informational) capture tous les événements importants (liaison down, connexion SSH, redémarrage) sans inonder le serveur de messages debug.
logging source-interface g0/0 / vlan 1
Pourquoi ? Définit l'interface dont l'IP sera utilisée comme adresse source des paquets Syslog. Permet au serveur de savoir quel équipement a envoyé le message. Routeur → G0/0 (IP LAN). Switch → VLAN 1 (IP de management).
logging buffered 64000
Pourquoi ? Stocke localement les 64 Ko de logs les plus récents en mémoire RAM de l'équipement. Utile pour consulter les logs en cas de perte de connectivité avec le serveur Syslog.
no logging console
Pourquoi ? Désactive l'affichage des logs sur la console. Sans ça, les messages de log interrompent les commandes tapées en console — très perturbant lors des configurations via PuTTY.
⌨ Différence Routeur vs Switch — Interface source
🔀 Routeur CRASH
logging source-interface g0/0
Le routeur envoie les logs depuis son interface LAN physique G0/0 — c'est cette IP qui apparaîtra comme source dans LogAnalyzer.
🔀 SwitchIntra / SwitchLigues / SwitchDMZ
logging source-interface vlan 1
Les switches n'ont pas d'interface physique avec IP — ils utilisent l'interface logique VLAN 1 (interface de management) comme source des logs.

Configuration VLAN 1 — IP de management des switches
Pourquoi configurer une IP sur VLAN 1 des switches ?

Un switch sans IP ne peut pas envoyer de paquets UDP Syslog vers le serveur — il n'a pas d'adresse source.

L'interface VLAN 1 est l'interface de management standard sur les switches Cisco — elle reçoit une IP pour permettre la communication IP du switch lui-même.

Capture 2 — Configuration IP sur VLAN 1 et activation
Switch — Interface management
! Configurer l'IP de management sur le VLAN 1
Switch(config)# interface vlan 1
Switch(config-if)# ip address 172.17.10.X 255.255.0.0
! Remplacer X par l'IP attribuée à chaque switch :
! SwitchIntra   → 172.17.10.21
! SwitchLigues  → 172.17.10.22
! SwitchDMZ     → 172.17.10.23

! Activer l'interface VLAN 1 (désactivée par défaut)
Switch(config-if)# no shutdown
Switch(config-if)# exit

! Définir la passerelle par défaut pour joindre le serveur Syslog
Switch(config)# ip default-gateway 172.17.0.1

Switch# write memory

! Tester la connectivité vers le serveur Syslog
Switch# ping 172.17.10.20
!!!!!  ← 5 points d'exclamation = ping réussi → switch joignable
⌨ Commande(s) utilisée(s)
interface vlan 1 / ip address / no shutdown / ip default-gateway
🧠 Pourquoi ip default-gateway sur un switch ?
Un switch de couche 2 ne fait pas de routage — il ne sait pas comment atteindre des réseaux différents. ip default-gateway lui dit "pour tout ce qui n'est pas ton réseau local, envoie au routeur". Sans ça, le switch peut pinger des machines du même VLAN mais pas le serveur Syslog s'il est sur un autre sous-réseau.
✅ Résultat attendu
Le switch a une IP de management — il peut envoyer des paquets UDP Syslog vers 172.17.10.20. Le ping vers le serveur doit retourner 5 !.
