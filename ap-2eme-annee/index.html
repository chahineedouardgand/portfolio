Mission 5 — Projet CRASH
Sécurisation du routeur avec des ACL
ACL Sécurité réseau Cisco IOS Filtrage
1
Contexte et objectif

Le routeur est un point critique du réseau : il assure la communication entre le réseau interne (LAN), la zone DMZ (serveurs) et Internet. Sans filtrage, n'importe quel trafic peut entrer ou sortir, ce qui représente un risque majeur pour la sécurité. Cette mission met en place des ACL (Listes de Contrôle d'Accès) pour autoriser ou bloquer les flux réseau selon des règles précises.

2
Principe de fonctionnement des ACL
  • Une ACL est une liste de règles appliquées sur une interface du routeur
  • Les règles sont lues de haut en bas : dès qu'une règle correspond, elle est appliquée
  • Chaque règle peut autoriser (permit) ou bloquer (deny) le trafic
  • Une règle implicite deny all est toujours présente à la fin — il faut donc ajouter explicitement les autorisations
3
Architecture réseau utilisée
Interfaces du routeur CRASH
G0/0LAN — 172.17.0.0/16
G0/1DMZ — 10.54.0.0/24
G0/2 / S0/0/0Internet — 192.168.230.1
4
Activation des logs avant configuration
Cisco IOS — Activation logs enable
conf t
service timestamps log datetime msec
logging buffered 64000
no logging console
end
write memory
Résultat

Le routeur CRASH est sécurisé avec des ACL : le réseau interne est protégé, les accès vers Internet sont contrôlés, et les tentatives interdites sont tracées dans les journaux Syslog.

Mission 7 — Projet CRASH
Mise en place d'un EtherChannel
EtherChannel LACP Agrégation de liens Redondance
1
Objectif

L'EtherChannel permet d'agréger plusieurs liens physiques en un seul lien logique pour améliorer la bande passante entre deux commutateurs, assurer une redondance de liens et améliorer la disponibilité des services réseau.

2
Principe de l'EtherChannel
  • Bande passante augmentée — plusieurs liens = plus de débit cumulé
  • Tolérance aux pannes — si un lien tombe, les autres prennent le relai
  • Répartition de charge — le trafic est distribué sur les ports agrégés
  • Simplification — un seul lien logique à configurer côté VLAN / Trunk
3
Configuration — Protocole LACP

Deux câbles Ethernet relient Fa0/1 et Fa0/2 entre les deux commutateurs. La configuration est identique sur les deux switches :

Cisco IOS — EtherChannel LACP interface range fa0/1 - 2
channel-group 1 mode active ! LACP actif
no shutdown
4
Vérification
Commande de vérification show etherchannel summary

Résultat attendu :
Po1(SU) → Port-channel actif
Fa0/1(P) Fa0/2(P) → Interfaces agrégées
Résultat

EtherChannel opérationnel — bande passante doublée entre les deux switches, infrastructure plus robuste et continuité de service assurée même en cas de panne d'un lien.

Mission 8 — Projet CRASH
Sécurisation des ports des commutateurs — Port Security
Port Security MAC address Sticky err-disabled
1
Pourquoi mettre en place le Port Security ?

Le Port Security protège les ports du switch contre les branchements non autorisés — PC inconnus, hubs, attaques de type MAC flooding. Dans l'infrastructure CRASH, chaque port utilisateur est limité à une seule adresse MAC. Si une autre machine est branchée, le port est automatiquement bloqué (err-disabled).

2
Configuration du Port Security
Cisco IOS — Port Security interface fastEthernet0/3
switchport mode access
switchport port-security
switchport port-security maximum 1 ! 1 seule MAC autorisée
switchport port-security mac-address sticky ! Apprend la MAC auto
switchport port-security violation shutdown ! Coupe en cas d'intrusion
3
Scénario de test — Détection d'intrusion
  • Une machine autorisée est branchée sur Fa0/3 → le switch apprend sa MAC (sticky)
  • Une autre machine est branchée → le switch détecte une MAC non autorisée
  • Le port passe automatiquement en état err-disabled → aucune communication possible
4
Remise en service du port
Remise en service interface fastEthernet0/3
shutdown ! Coupe manuellement
no shutdown ! Réactive — après vérification !
Résultat

Le Port Security est fonctionnel — tout branchement non autorisé est immédiatement détecté et bloqué. La sécurité physique de l'infrastructure CRASH est assurée.

Étape 4 — DHCP sur HAProxy (isc-dhcp-server)

Étape 5 — HAProxy (répartition de charge + HTTPS)
HAProxy — /etc/haproxy/haproxy.cfg # Frontend HTTP → port 80
frontend http_front
  bind *:80
  default_backend web_servers

# Frontend HTTPS → port 443 (après création du .pem)
frontend https_front
  bind *:443 ssl crt /root/ssl/haproxy.pem
  default_backend web_servers

# Backend — roundrobin entre Web1 et Web2
backend web_servers
  balance roundrobin
  option httpchk GET /
  server web1 10.0.10.12:80 check
  server web2 10.0.10.13:80 check
8
Étape 6 — Certificat TLS + DNS Technitium
Certificats TLS — Création AC + Certificat site # 1. Créer l'AC (Autorité de Certification)
openssl req -x509 -newkey rsa:2048 -keyout ca.key -out ca.crt -days 365 -nodes -subj "/CN=AC"

# 2. Créer la clé + CSR du site
openssl req -newkey rsa:2048 -nodes -keyout site.key -out site.csr -subj "/CN=monsite.local"

# 3. Signer avec l'AC
openssl x509 -req -in site.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out site.crt -days 365

# 4. Créer le .pem pour HAProxy
cat site.crt site.key ca.crt > haproxy.pem

DNS Technitium est lancé via Docker sur HAProxy (port 5380 pour l'interface web, port 53 pour les requêtes). Une zone est créée avec un enregistrement A pointant vers l'IP de HAProxy.
